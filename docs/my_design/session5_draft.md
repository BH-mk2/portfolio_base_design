## 1. 非機能要件・性能要件

### 1.1 性能・リソース要件
【プレビューのフレームレート、API応答遅延、メモリ/CPU使用率の制限値などを記述してください。】

- **フレームレート・解像度サポート**:
  - 最大解像度: 4K (3840x2160)、フレームレート: 24fps のストリーム受信を、コマ落ちなく処理できること。
- **プレビュー遅延 (レイテンシ)**:
  - カメラが画像を送信してから `PreviewCallback` が呼び出されるまでのSDK内部処理遅延を、平均10ms以下に抑えること。
- **CPU・メモリ使用率**:
  - 画像ストリーム処理中のSDK追加のメモリフットプリントを、最大100MB以下に抑えること（バッファプールを含む）。
  - シングルカメラ接続時のSDKスレッド全体(プレビュー用スレッド除く)のCPU使用率を、一般的なCore i5環境で5%以下に抑えること。
- **同時接続数**:
  - 同一PC上で最大128台のカメラを同時接続・制御できること。

### 1.2 動作環境・互換性要件
【サポートするOS、コンパイラ、ハードウェア要件などを記述してください。】

- **対応OS**:
  - Windows 11 (64bit)
  - Linux (Ubuntu 20.04/22.04 LTS, x86_64)
  - macOS 15,26,27 (arm64)
- **コンパイラ・言語規格**:
  - C++17規格で内部実装され、C言語のインターフェース（Cリンケージ）を提供する。
- **通信プロトコル規格**:
  - PTP (Picture Transfer Protocol) over USB (MTP拡張を含む)、および PTP over IP (GigE準拠)。

## 2. テスト観点

### 2.1 単体テスト方針 (Unit Test)
【API単体の機能や状態遷移、メモリリークをどうテストするか記述してください。】

- **状態遷移テスト**:
  - 各接続状態（Uninitialized, Initialized, Connecting, Connected, Streaming 等）において、無効なAPI呼び出し時のエラー返却テスト。
- **バッファプール管理テスト**:
  - アプリ側から渡された複数の画像バッファの循環利用テスト、バッファサイズ不足時のエラー検出テスト。
- **メモリリーク検出**:
  - 各API（特に接続・切断、プレビュー開始・停止）の1万回連続呼び出しによるメモリリークの自動検出（ValgrindやMSVCのリーク検出機能を使用）。

### 2.2 結合・システムテスト方針 (Integration Test)
【ハードウェアを含めた実機テスト、通信エラー時のエラーリカバリ、負荷テストの方針を記述してください。】

- **実機検証テスト**:
  - 対応予定の代表カメラモデル（モデルA、モデルB等）を用いたAPI正常動作テスト。
- **異常系リカバリテスト**:
  - ストリーミング中の物理的USBケーブル抜去、LANケーブル切断時の `EventCallback`（`EVENT_TYPE_DEVICE_MISSING`）発火確認。
  - 再接続設定ON時の自動復旧およびOFF時の即時 `Disconnected` 遷移確認。
- **マルチカメラ負荷テスト**:
  - 限界接続台数（8台）のカメラから同時に4K/30fpsで画像を受信し、24時間連続動作させた際の安定性（ハングアップ、メモリ増加がないこと）の検証。

### 2.3 カメラシミュレータ (Mock) の活用方針
【実機がない開発・テスト環境において、どのようにSDKの動作検証を行うか記述してください。】

- **ソフトウェアシミュレータの作成**:
  - PTP通信部をモック（Mock）化し、仮想的なカメラ応答や画像ストリーム（テストパターン画像等）を生成する「仮想カメラシミュレータ」を内部ビルド用に用意する。
  - これにより、実機が手元にないCI（継続的インテグレーション）サーバー上での自動テスト走行を可能にする。
