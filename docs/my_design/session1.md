# C社カメラ用SDK 基本設計書 - Session 1

---

## 1. 機能概要

### 1.1 開発の目的・背景
- 本SDKは、C社製カメラを制御するアプリケーションのセットアップを簡易化し、シビアなタイミングでの自動制御を容易に実現することを目的とする。
- 物理カメラの接続管理、独自プロトコル（PTP over USB/IP）の制御、および画像ストリーミングの受信といった低レイヤ通信や同期スレッド処理は開発コストが高く、実装上の不具合（デッドロックやメモリリーク等）も起きやすい。
- 本SDKがこれらの低レイヤ制御を完全にカプセル化（隠蔽）し、シンプルかつ統一的なCリンケージAPIとして提供することで、上位アプリケーション開発者が画像解析や製品検査システムなどのビジネスロジックの開発に専念できるようにする。

### 1.2 主な機能 (In Scope)
- カメラデバイスの自動検出と接続・切断
- カメラ設定の取得と変更（解像度、フレームレート、露光時間、ゲイン等）
- ライブプレビュー用のフレームデータストリーミング
- カメラへのリモート操作（シャッター、AF、録画、メニュー操作等）
- カメラ内のコンテンツ取得・変更・削除

### 1.3 対象外とする機能 (Out of Scope)
- 画面描画機能
  - クロスプラットフォーム展開を考慮し、各プラットフォームやフレームワーク（WPF, Unity等）の最適な描画機能に委ねることで、SDK自体をポータブルで軽量にするため。
- 画像・動画ファイルの保存機能
  - アプリケーションにファイル出力用コールバックを提供しつつも、ファイルIO自体はSDK内で行わない。これにより、二重のファイル書き込み処理によるパフォーマンス低下を防ぎ、メモリ上で直接画像データを扱いたいユースケースにも対応できるようにするため。
- カメラ以外との通信制御
  - クラウドやバックエンドサーバーとの通信はアプリケーション側の役割とし、本SDKはカメラとの接続制御に専念するため。
- カメラ映像内のピクセル解析処理
  - 被写体検出やAF・フォーカスなどの処理は可能な限りカメラデバイス側のハードウェア機能を利用し、SDKはカメラから得られた結果構造体をAPIとして上位アプリに渡すのみとする。画像処理ライブラリやAIモデルを用いたピクセル解析はSDKの範疇を超えるため対象外とする。

### 1.4 【設計判断】なぜこの機能範囲（スコープ）にしたのか？
- **[In Scope] の判断理由**
  - **自動検出、接続・切断、リモート操作、コンテンツ操作**:
    - これらの機能はすべてカメラ制御に必須となる主要要件である。これらにはメーカー固有のPTPコマンド仕様や接続プロトコルが含まれており、機密保持（仕様の非公開化）と実装難易度の観点から、外部に直接公開するのではなくSDK内部にカプセル化して隠蔽すべきと判断した。
  - **ライブプレビュー用のフレームデータストリーミング**:
    - アプリケーション側で直接通信ソケットやUSBエンドポイントから画像を受信しようとすると、SDK内のコマンドポーリング通信と衝突（競合）が発生する。これを防ぎ、通信の排他制御を一元管理するためにSDKの内部機能として実装した。
- **[Out of Scope] の判断理由**
  - **画面描画機能、ファイル保存機能**:
    - 特定のOSやUIフレームワーク、ディスクIO処理に密結合するのを防ぐため。SDKをプラットフォーム非依存の純粋な「通信・制御ライブラリ」とすることで、Windows、Linux、macOSなどの多様な環境へ少ない変更コストで展開できるようになる。
  - **ピクセル解析処理、外部通信制御**:
    - SDKの「カメラの制御および画像データの伝送」という本来の関心事から外れるため。これらの機能を排除し「関心の分離」を徹底することで、SDKの肥大化を防ぎ、保守性・テスト容易性を高く維持している。

## 2. ユースケース一覧

### 2.1 アクターの定義
- **アプリ（開発者）**: SDKのAPIを呼び出し、カメラを制御する主体。
- **C社製カメラ**: SDKが直接制御する物理ハードウェア。

### 2.2 ユースケース一覧表

| ID | ユースケース名 | 概要 | アクター | 区分 |
| :--- | :--- | :--- | :--- | :--- |
| UC-01 | カメラを検出する | PCに接続されているC社製カメラの一覧を取得する。 | アプリ | 正常 |
| UC-02 | カメラに接続する | 指定したカメラとの通信経路を確立し、制御可能な状態にする。 | アプリ、カメラ | 正常 |
| UC-03 | カメラから切断する | カメラとの通信を安全に切断し、リソースを解放する。 | アプリ、カメラ | 正常 |
| UC-04 | プレビューを開始する | カメラから連続的な画像データの取得を開始し、アプリに通知する。 | アプリ、カメラ | 正常 |
| UC-05 | プレビューを停止する | 画像データのストリーミングを停止し、受信スレッドやバッファを休止する。 | アプリ | 正常 |
| UC-06 | パラメータを取得する | カメラの現在の設定（露出、ゲイン、解像度等）を取得する。 | アプリ、カメラ | 正常 |
| UC-07 | パラメータを変更する | 露光時間やゲインなどの設定値をカメラに反映する。 | アプリ、カメラ | 正常 |
| UC-08 | 操作コードを送る | シャッター、AF、録画の開始・停止など、カメラの物理動作を指示する。 | アプリ、カメラ | 正常 |
| UC-09 | コンテンツ一覧を取得する | カメラ内のストレージに保存されている静止画・動画ファイルの一覧を取得する。 | アプリ、カメラ | 正常 |
| UC-10 | コンテンツをダウンロードする | 指定したファイルをカメラからPCへ転送（ダウンロード）する。 | アプリ、カメラ | 正常 |
| UC-11 | コンテンツを削除する | カメラ内の不要なコンテンツを削除し、ストレージ容量を解放する。 | アプリ、カメラ | 正常 |

| UC-13 | プレビュー中に通信切断 | ストリーミング中に物理的なケーブル抜け等で通信が途絶する。 | カメラ | 異常 |

### 2.3 【設計判断】なぜこのユースケース構成にしたのか？
- **正常系のペア・対称性の確保 (UC-04/05, UC-06/07)**:
  - プレビューの「開始」に対する「停止」、パラメータの「変更」に対する「取得」など、API（`StartPreview`/`StopPreview` や `SetCameraSetting`/`GetCameraSetting`）として対になる動作をユースケース段階から定義した。これにより、API設計や内部状態遷移（`Streaming` 状態から `Connected` 状態への復帰等）に矛盾が生じるのを防ぎ、一貫したインターフェース設計を行うため。
- **コンテンツ管理ユースケースの明文化 (UC-09/10/11)**:
  - In Scopeに定義された「カメラ内のコンテンツ取得・変更・削除」を実現するため、必要な動作を「一覧取得」「ダウンロード」「削除」として具体的に定義した。これにより、PTPのファイル転送モード時の挙動やコマンド仕様の設計漏れを防ぐ。
- **初期段階からの異常系の定義 (UC-12, 13)**:
  - 物理ハードウェアやネットワークを介した通信では「接続失敗」や「動作中のケーブル切断」が頻繁に発生する。これらを初期から定義することで、エラーコード体系（Session 4）やコールバックによる非同期エラー通知（`EventCallback`）の設計が後手になるのを防ぎ、耐障害性の高い堅牢なSDKを設計するため。
